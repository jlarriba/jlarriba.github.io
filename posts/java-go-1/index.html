Sun, Go and Java

13.06.2020 15:52

This week has been sunny and hot down here, which is something after 2 years in the Netherlands I still are not very used to. I have been thinking a lot about programming languages (conclusions might appear on this very blog in the future).

I grew up as a developer programming in Java. I have dedicated countless hours to the Java programming language. I also dedicated some time to Ruby (on Rails) back in 2008. It was really fun.

I programmed A LOT in Javascript back in my time, before Javascript was considered a “serious” development language. We were doing Single Page Applications with jQuery, sweat and blood. There was also that nice time where I was developing 3D in Javascript with WebGL.

More recently, since about three years, I have been programming in Go. As Go has never been the language of choice of the companies I worked for, I was never able to work on really big projects, as I did with Java, but only small system-things related to some site reliability engineer. One small app that kills a certain process, a Prometheus exporter for some kind of specialized hardware, you know.

I have been comparing in my mind Java and Go, and the conclusions are kind of counter-intuitive. This happens almost every time you spend some time thinking thourougly about everything: your opinion might not be the same as the majority of Twitter’s and thus it is “counter-intuitive”.

I definitively love Go. It is a cute little language which generates native binaries, does not have Object Oriented Programming, it is very easy to start with, is statically-typed, can be run as script and it is garbage-collected. It is REALLY nice when you come from what we can call and “enterprise java developer”. It also can be written using nice CLI tools and a text editor.

The services created with Go are small, with no memory overhead and they start really fast. I do not foresee leaving Go in the following future. I hope I get the opportunity to work in a big go-related project in some moment.

However, liking Go does not make me hate Java. I hate the Java/Spring/Hibernate shit that is programmed (still with eclipse, most of the time) in the big IT consultancies. The prevalence of this enterprise applications, developed by hordes of mostly untrained and junior programmers, memory hogs and utterly slow, permeate the popular culture and make some twitter gurus and fanboys to reach the absurd axiom: Java is slow.

Moreover, most of the people I have seen complaining about the slowness of Java suggest replacing it with… Python!! The Python web development framework Django created a lot of this kind of gremlins.

As you might know, my fellow reader, Java is not slow at all. Of course is slower than C or C++, and of course it has the JVM memory overhead, but the HotSpot JIT compiler is so optimized now that it is crazy.

If you dig into the Java (or JVM, there are a lot of other languages to choose from) world, you realize that for most of things Java, and the OO programming model are still very good. You always need to avoid the enterprisey-frameworks (mostly Spring and its nonsense reflection everywhere policy) and release your threads!

        myThread.start();
        Runtime.getRuntime().addShutdownHook(new Thread(
                () -> {
                    logger.info("Shutdown hook");
                    ((ConsumerRunnable) myConsumerRunnable).shutdown();
                }
        ));
